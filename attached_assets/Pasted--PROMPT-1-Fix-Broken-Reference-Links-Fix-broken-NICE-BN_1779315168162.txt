## PROMPT 3 — Build the Collapsible Revision Panel (with guaranteed working hyperlinks)

> *Build a collapsible "Revision Panel" under every question in my medical question bank, with guaranteed working hyperlinks throughout.*
> 
>*Hyperlinking rules — follow these strictly:*
> - Every reference must be a clickable <a href="..." target="_blank" rel="noopener noreferrer"> link
> - Only use top-level stable URLs:
>   - NICE Guidelines → https://www.nice.org.uk/guidance/[guideline-id]
>   - NICE CKS → https://cks.nice.org.uk/topics/[topic-name]/
> - Never construct #anchor deep links — NICE anchors are unpredictable and will 404
> - Instead, state the specific section in plain text next to the link, e.g. "See Section 1.2 — Pharmacological management"
> - Before saving any URL, strip out any /treatment-summaries/ path and replace with the correct CKS equivalent
> - Validate every URL follows exactly one of the two formats above
> 
>*Panel design:*
> - Single cohesive card below the answer explanation, above the reference links
> - Dark header bar labelled "📚 Revision Panel" in white text with a chevron icon
> - Collapsed by default, expands on tap
> - Two rows inside with a subtle dividing line:
>   - *📖 What to Read* — amber/gold left border
>   - *⚠️ Common Mistake* — red/coral left border
> - Rounded corners, soft shadow, fully mobile-friendly
> 
>*📖 What to Read row must contain:*
> - A working hyperlink to the top-level NICE guideline or CKS topic page, labelled with the guideline name
> - In plain text below the link: the exact section/chapter the student should navigate to
> - One sentence explaining what concept that section covers, tied directly to what this question is testing
> 
>Example format:
> > [NICE NG80 — Asthma](https://www.nice.org.uk/guidance/ng80)
> > Navigate to: Section 1.2 — Pharmacological treatment, Step 3 therapy
> > This section covers when to add LABA to ICS — the step-up decision tested in this question
> 
>*⚠️ Common Mistake row must contain:*
> - The specific wrong answer most candidates pick, named explicitly
> - One sentence explaining the exact reasoning error
> - The keyword or clue from the question stem that gives away the correct answer, shown in bold
> 
>Example format:
> > Most candidates pick Salbutamol PRN — this misses that the question describes uncontrolled chronic asthma, not an acute attack
> > The clue is *"despite regular preventer inhaler"* — this phrase signals step-up therapy is needed
> 
>*Final rules:*
> - Every question gets exactly one Revision Panel
> - All content must be specific to that question — no generic condition-level text
> - Remove any previously added standalone "What to Read" or "Common Mistake" callouts and consolidate into this panel
> - Do not change question content, answer options, explanations, or existing reference links
> - Collapsed state shows just the "📚 Revision Panel" header with a subtle chevron icon

## PROMPT 4 — Automatic Link Checker Script

> *Write a link-checking script that automatically tests every reference URL in my medical question bank and flags any that are broken.*
> 
>*What the script should do:*
> 
>1. *Extract every URL* from the references and Revision Panel sections across all questions
> 
>2. *Test each URL* by sending a HEAD request and checking the HTTP response code:
>    - ✅ 200 — working, ignore
>    - ⚠️ 301/302 — redirected, log the new destination URL
>    - ❌ 404 — broken, flag immediately
>    - ❌ 403/500+ — unreachable, flag immediately
>    - ❌ Timeout — flag as unreachable
> 
>3. *Generate a report* saved as link-report.json containing:
>    - Total links checked
>    - Total passing ✅
>    - Total broken ❌
>    - For each broken link: the question ID, question title, broken URL, and error code
> 
>4. *Auto-fix where possible:*
>    - Any URL containing /treatment-summaries/ → replace with cks.nice.org.uk/topics/ equivalent
>    - Any URL returning a 301 redirect → update the stored URL to the final destination
>    - Log every auto-fix made
> 
>5. *Display a summary panel in the app* (visible only in admin/dev mode) showing:
>    - A green banner if all links pass
>    - A red banner with a count if any are broken, with a button to download link-report.json
> 
>*Technical requirements:*
> - Use node-fetch or the built-in fetch API — no external paid services
> - Run checks in batches of 5 at a time to avoid rate limiting NICE servers
> - Add a 1 second delay between batches
> - Runnable via npm run check-links or triggered automatically on app startup in development mode only
> - Do not run link checks in production on every page load — only on demand
